# Code-Based, Programmatic, and Design-as-Code Approaches to Electronics and PCB Design: A Review

## 1. Introduction

The convergence of software engineering practices with electronics design has given rise to a family of tools and frameworks that treat printed circuit board (PCB) schematics, netlists, and layouts as code artifacts amenable to version control, continuous integration (CI), and reproducible builds. Analogous to how CadQuery and OpenSCAD enable "code-CAD" for mechanical design, board-level hardware description languages (HDLs) and programmatic EDA tools promise to bring generator-based abstractions, type systems, parametric reuse, and automated verification to electronics (lin2020opportunitiesandchallenges pages 1-2). Lin and Hartmann framed this space by noting that board-level HDLs can encapsulate specialized knowledge in reusable libraries, apply object-oriented type hierarchies with voltage and current parameters for correctness checks, and use generator constructs that automatically size components per application—analogous to the Chisel approach in chip design (lin2020opportunitiesandchallenges pages 1-2, lin2020opportunitiesandchallenges pages 4-4). This review surveys the major tools and frameworks in this space, discusses LLM-based generative approaches, and assesses fitness for an open-hardware, CI-driven workflow suitable for a modular multi-powder dosing device.

### 2.1 atopile (.ato Language)

Atopile (github.com/atopile/atopile) introduces a custom `.ato` hardware description language compiled by a Python-based toolchain into KiCad project files. The programming model emphasizes textual, declarative circuit descriptions with module imports, parameterized components, and hierarchical composition. Designs are expressed in `.ato` files that compile to KiCad schematics and netlists; Gerber and BOM generation is delegated downstream to KiCad. The text-first nature makes atopile exceptionally Git-friendly, and its package manager supports reusable module libraries. DRC is handled by KiCad after compilation. Licensed under Apache-2.0, atopile is early-stage but represents one of the most complete "design-as-code" visions for electronics, with an active community and growing component ecosystem. Its philosophy aligns closely with the broader vision of board-level HDLs that raise abstraction while preserving fine-grained control (lin2020opportunitiesandchallenges pages 4-4).

### 2.2 tscircuit (React-for-PCB)

tscircuit (github.com/tscircuit/tscircuit) models PCB designs as React/TypeScript component trees, where JSX elements describe resistors, capacitors, ICs, traces, and board outlines. This leverages the entire npm/TypeScript ecosystem for composition, testing, and CI. Circuits are defined declaratively using component props for values and footprints, then rendered to Gerber files, BOMs, netlists, and 3D previews. The React model naturally supports parametric reuse through props, higher-order components, and package publishing. DRC is handled internally with growing capability. Licensed under MIT, tscircuit is rapidly evolving and particularly appealing to web-developer communities, though its electronics library coverage is still growing.

### 2.3 SKiDL

SKiDL (github.com/devbisme/skidl) is a mature Python embedded DSL for schematic capture where designers instantiate `Part` objects, create `Net` connections, and call built-in ERC checks—all in Python (zou2602pcbschemagenconstraintguidedschematic pages 4-5, gutierrez2023pythonframeworkfor pages 2-3). Gutiérrez et al. noted that SKiDL and PySpice are Python tools focused on PCB design and netlist generation, though with some simulator limitations (gutierrez2023pythonframeworkfor pages 2-3). SKiDL's strength lies in leveraging full Python for loops, functions, classes, and conditional logic to generate parameterized, reusable circuit descriptions. It outputs netlists compatible with KiCad and includes built-in electrical rule checking. The PCBSchemaGen system specifically chose SKiDL as its intermediate representation because "Python-based hardware-description languages have high information density and match LLM fine-tuning" (zou2602pcbschemagenconstraintguidedschematic pages 4-5, zou2602pcbschemagenconstraintguidedschematic pages 3-4). SKiDL is MIT-licensed, widely cited in research, and represents the most mature purely code-based schematic capture tool.

### 2.4 BHDL (Board HDL)

BHDL is a declarative embedded DSL implemented in Racket, presented at DAC 2021, where every circuit element—even a resistor—is modeled as a function to enable hierarchical, modular composition (li2021bhdlalucid pages 1-2). BHDL provides concise connection syntax including `net`, `series`, `parallel`, and `bus` forms, plus layout combinators (`hstack`, `vstack`, `rotate`, `at`) inspired by VLSI placement tools (li2021bhdlalucid pages 2-3, li2021bhdlalucid pages 3-5). For layout, BHDL integrates an analytical global placer based on RePlace and a simulated-annealing detailed placer with rotation support, implemented in Julia (li2021bhdlalucid pages 5-6). The system exports KiCad-format PCB files, Gerber, and BOMs, and supports REPL-style development via a Jupyter kernel (li2021bhdlalucid pages 1-2, li2021bhdlalucid pages 5-6). As a research prototype, BHDL demonstrates the power of functional composition for PCB design but has a smaller ecosystem than Python-based alternatives.

### 2.5 Polymorphic Blocks

Polymorphic Blocks, developed by Lin et al. (2020), is a Python-embedded HDL that extends hierarchical block diagrams with polymorphism, typed ports carrying electrical parameters (voltage, current, thresholds), and generator blocks that programmatically instantiate subcircuits (lin2020opportunitiesandchallenges pages 1-2, lin2020polymorphicblocksunifying pages 5-7, lin2020polymorphicblocksunifying pages 2-4). Ports and links propagate parameters through a directed constraint graph and include assertions for correctness checking; generators can access solved values to automatically size components such as resistors from the E24 series (lin2021humancenteredcircuitboard pages 54-56, lin2021humancenteredcircuitboard pages 56-60, lin2020polymorphicblocksunifying pages 4-5). The compiler lowers designs via refinement and generation steps and exports netlists for KiCad import, with deterministic component naming to support layout updates (lin2020polymorphicblocksunifying pages 5-7). The project is open-source on GitHub and demonstrates how type systems and constraint propagation can provide meaningful automation for intermediate-level designers (lin2020polymorphicblocksunifying pages 5-7, lin2020polymorphicblocksunifying pages 10-11).

### 2.6 JITX (Stanza-Based Programmatic PCB)

JITX (jitx.com) is a commercial programmatic PCB design platform built on the Stanza programming language. Referenced in multiple academic works as JITPCB (lin2021humancenteredcircuitboard pages 27-30, li2021bhdlalucid pages 6-6), JITX embeds circuit construction primitives in a generator-style programming model, enabling parameterized design families with algorithmic placement, routing, and constraint-driven automation. It produces complete manufacturing outputs including Gerber files, BOMs, and pick-and-place data. JITX's integrated rule and constraint system is among the most capable code-native PCB verification flows. However, its proprietary licensing creates friction for open-hardware workflows, and the requirement to learn Stanza represents a barrier compared with Python-based alternatives (guardado2023“sudomakeme pages 129-132). JITX is commercially mature and targets professional engineering teams requiring high automation.

### 2.7 PCBmodE

PCBmodE (github.com/boldport/pcbmode) takes a unique approach by defining PCB geometry and artwork through JSON configuration files and SVG graphics. This makes it excellent for artistic or geometry-driven board designs where visual aesthetics matter. It produces Gerber manufacturing files from SVG-based board descriptions. However, its electrical abstraction is weaker than HDL-style tools—it lacks netlist-level schematic capture, DRC, or ERC capabilities. The project has seen limited recent activity but remains interesting as a purely text/data-driven PCB definition approach that is naturally Git-friendly.

### 2.8 gEDA/gaf

The gEDA project provides a traditional Unix-style open-source EDA suite with text-based schematics (gschem), scriptable netlisting utilities, and the PCB layout tool with Gerber export via gerbv. Files are text-based and CLI tools support scripted workflows, making gEDA compatible with version control and CI pipelines. However, the toolchain feels legacy compared with modern alternatives, and parametric design support is less ergonomic than newer HDL-based flows. Licensed under GPL, gEDA remains functional but has largely been superseded by KiCad for new open-hardware projects.

### 2.9 Horizon EDA

Horizon EDA (github.com/horizon-eda/horizon) is a GUI-first open-source EDA tool with strong DRC, manufacturing-output capabilities, and structured text-backed project data. While not a "code-CAD" tool per se, its project files are Git-manageable and it supports some scriptability and CLI-driven export. Horizon EDA excels at part and library management and is licensed under GPL-3.0. It bridges the gap between GUI-first design and version-controllable workflows.

### 2.10 Edea

Edea (github.com/edea-dev/edea) is a Python library for programmatically generating or manipulating KiCad project files. It operates as a tooling layer rather than a full EDA suite, enabling scripts to create, modify, or template KiCad schematics and PCB layouts. This makes it ideal for CI pipeline integration—for example, automated variant generation or BOM manipulation—while delegating ERC/DRC to KiCad itself. Edea is open-source and fits well into workflows where KiCad is the primary design tool but automation is needed around it.

### 2.11 KiCad Python Scripting and kicad-cli

KiCad, the dominant open-source PCB tool (GPL-3.0), offers extensive automation through its Python scripting API, action plugins, and the newer `kicad-cli` command-line interface. The Python API enables programmatic creation and manipulation of schematics, footprints, and board layouts; action plugins extend the GUI with custom functionality (lin2021humancenteredcircuitboarda pages 97-100, lin2021humancenteredcircuitboard pages 97-100). The `kicad-cli` tool (introduced in KiCad 7+) supports headless export of Gerber files, drill files, BOMs, PDFs, STEP models, and schematic/PCB renderings—making it the key enabler for CI pipelines. KiCad's text-based file formats (`.kicad_sch`, `.kicad_pcb`) are Git-manageable, though diffs can be complex (stirling2022hardopsutilisingthe pages 7-9). The HardOps framework demonstrated that CI pipelines can automate quality checks, artifact generation, and documentation for open-hardware projects, with KiCad as a central tool (stirling2022hardopsutilisingthe pages 7-9, stirling2022hardopsutilisingthe pages 3-4, stirling2022hardopsutilisingthe pages 9-12).

**PCBSchemaGen** (Zou et al.) is a training-free, constraint-guided framework that uses LLMs to generate SKiDL Python code from natural-language specifications. It builds a knowledge graph from IC datasheets encoding pin-role semantics (36 role types) and topological constraints, compressing per-device context from ~16k tokens to ~300 tokens (zou2602pcbschemagenconstraintguidedschematic pages 4-5, zou2602pcbschemagenconstraintguidedschematic pages 3-4). Multi-stage verification combines SKiDL syntax/ERC checks with topology verification via the VF2 subgraph isomorphism algorithm (zou2602pcbschemagenconstraintguidedschematic pages 5-6). On a 23-task benchmark spanning digital, analog, and power domains, PCBSchemaGen achieved 93.3% success on easy tasks and 78.1% on hard tasks, with a ~37× speedup over human designers (zou2602pcbschemagenconstraintguidedschematic pages 7-8, zou2602pcbschemagenconstraintguidedschematic pages 6-7, zou2602pcbschemagenconstraintguidedschematic pages 1-2).

**HWE-Bench** (Qiu et al.) provides a more rigorous evaluation with 300 real-world tasks across 8 domains paired with 2,914 IC datasheets. It applies dual verification: static electrical rule checking and dynamic LTspice transient simulation (qiu2603hwebenchcanlanguage pages 5-6, qiu2603hwebenchcanlanguage pages 1-2, qiu2603hwebenchcanlanguage pages 4-5). The results are sobering: even the best model (Claude Sonnet 4.5) achieved only 8.15% overall correct-design pass rate, despite average static pass rates of ~72% (qiu2603hwebenchcanlanguage pages 5-6, qiu2603hwebenchcanlanguage pages 1-2). Key failure modes include context overload with large component lists, MCU pin-multiplexing errors, and inability to maintain global logical consistency (qiu2603hwebenchcanlanguage pages 6-8, qiu2603hwebenchcanlanguage pages 8-8).

**PCEval** (Song et al.) benchmarks LLMs on physical computing tasks combining circuit and code generation for Arduino projects. Models performed reasonably on logical circuit design and code generation but struggled with physical breadboard-level layout, particularly pin-conflict avoidance (song2601pcevalabenchmark pages 1-3).

These results indicate that while LLM-assisted schematic generation shows promise for constrained subtasks—especially when augmented with knowledge graphs, iterative feedback, and domain-specific verification—full end-to-end autonomous PCB design remains unreliable for production use.

## 4. Trade-offs: GUI-Based EDA vs. Code-Based EDA vs. AI/Generative EDA

**GUI-based EDA** (KiCad, Altium, EAGLE) excels at interactive layout optimization, complex trace routing, and visual inspection. Experienced designers achieve superior placement and routing quality through direct manipulation. However, GUI tools produce binary or semi-structured files that resist meaningful Git diffs, make parametric variant generation manual, and are difficult to drive from CI pipelines without significant scripting effort (stirling2022hardopsutilisingthe pages 7-9, stirling2022hardopsutilisingthe pages 12-13).

**Code-based EDA** (atopile, SKiDL, tscircuit, BHDL, JITX) treats electronics design as a programming activity with all attendant benefits: plain-text source control, meaningful diffs and code review, automated testing, parameterized reuse through functions and modules, and deterministic CI-generated outputs (lin2020opportunitiesandchallenges pages 1-2, stirling2022hardopsutilisingthe pages 7-9). The trade-off is that placement and routing often require handoff to downstream tools (KiCad, FreeRouting) or are handled by less mature auto-placers (li2021bhdlalucid pages 5-6). Lin and Hartmann identified confidence in code-generated hardware as a key challenge: unlike software, hardware re-spins are costly, so community mechanisms for marking subcircuits as "physically fabricated and tested" are important (lin2020opportunitiesandchallenges pages 4-4).

**AI/Generative EDA** offers the most ambitious vision—natural-language to manufactured board—but current capabilities fall short. The HWE-Bench results (8.15% full-design pass rate) demonstrate that LLMs lack the physical intuition and global consistency needed for reliable circuit design (qiu2603hwebenchcanlanguage pages 5-6, qiu2603hwebenchcanlanguage pages 6-8). Targeted applications such as subcircuit suggestion, component selection, or datasheet extraction are more immediately practical (lin2020polymorphicblocksunifying pages 10-11, guardado2023“sudomakeme pages 20-24).

## 5. Recommendations for an Open-Hardware, CI-Rendered Modular Dosing Device

For a modular multi-powder dosing device—likely involving motor drivers, load-cell amplifiers, microcontroller modules, and communication interfaces as repeated building blocks—the following architecture is recommended:

**Primary design tool: KiCad + atopile or SKiDL for schematic generation.** atopile's `.ato` files provide the cleanest "design-as-code" experience with native KiCad integration, Git-friendliness, and a growing package ecosystem. SKiDL offers a mature, well-documented alternative with the advantage of full Python ecosystem access and demonstrated compatibility with LLM-assisted generation workflows (zou2602pcbschemagenconstraintguidedschematic pages 4-5, zou2602pcbschemagenconstraintguidedschematic pages 2-3). Both produce KiCad-compatible outputs.

**CI pipeline: kicad-cli + GitHub/GitLab CI.** Following HardOps principles (stirling2022hardopsutilisingthe pages 7-9, stirling2022hardopsutilisingthe pages 3-4, stirling2022hardopsutilisingthe pages 9-12), the pipeline should run ERC/DRC checks, generate Gerber/drill files, render board previews (SVG/PDF), produce BOMs, and publish release artifacts on each commit. The `kicad-cli` tool enables all of these operations headlessly. Docker containers with KiCad pre-installed ensure reproducible builds.

**Modular reuse:** The dosing device's repeated channel modules (each with motor driver, load cell, valve control) should be defined as parameterized blocks—either atopile modules or SKiDL generator functions—instantiated N times with channel-specific parameters. Polymorphic Blocks' approach of abstract block types with typed ports and constraint propagation provides a strong conceptual model for this (lin2021humancenteredcircuitboard pages 54-56, lin2020polymorphicblocksunifying pages 4-5).

**Edea** can serve as a CI-layer tool for KiCad file manipulation tasks such as variant generation, BOM post-processing, or automated schematic transformations.

**Layout:** For a production device, interactive KiCad layout with replication of channel modules (using hierarchical sheets and "connection room" replication) remains preferable to fully automated placement, which is not yet sufficiently mature in open-source tools for mixed-signal designs (li2021bhdlalucid pages 5-6).

**Licensing alignment:** atopile (Apache-2.0), SKiDL (MIT), KiCad (GPL-3.0), Edea (open-source), and tscircuit (MIT) all support open-hardware distribution. JITX's proprietary model is less suitable for community-driven projects.

## 6. Conclusion

The design-as-code paradigm for electronics has matured substantially since 2020, with tools spanning custom HDLs (atopile), Python eDSLs (SKiDL, Polymorphic Blocks), functional languages (BHDL in Racket), React/TypeScript frameworks (tscircuit), and commercial platforms (JITX). These tools share the vision of raising abstraction through generators, type systems, and reusable libraries while producing version-controllable, CI-amenable artifacts (lin2020opportunitiesandchallenges pages 1-2, li2021bhdlalucid pages 1-2). LLM-based approaches show promise for constrained subtasks but remain unreliable for full-design autonomy (qiu2603hwebenchcanlanguage pages 5-6, zou2602pcbschemagenconstraintguidedschematic pages 1-2). For an open-hardware modular dosing device, a hybrid workflow combining code-generated schematics (atopile or SKiDL), KiCad-based layout/DRC, and CI automation via `kicad-cli` and HardOps principles offers the best balance of reproducibility, manufacturability, and community accessibility.
