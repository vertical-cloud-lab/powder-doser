# 05 — Mitigating LLM spatial-reasoning and assembly failures in code-based CAD

**Status:** Synthesis triggered by [PR #34 review comment 4433787200](https://github.com/vertical-cloud-lab/powder-doser/pull/34#issuecomment-4433787200) (@sgbaird-alt) on the spatial-reasoning struggles observed in [PR #35 review #4274628757](https://github.com/vertical-cloud-lab/powder-doser/pull/35#pullrequestreview-4274628757) (@williamulbz on a Claude Opus 4.7-generated CadQuery model). It pulls together the relevant material already collected in [`04-generative-cad-academic-literature.md`](04-generative-cad-academic-literature.md) and [`03-generative-cad-landscape.md`](03-generative-cad-landscape.md) and is augmented by a follow-up Edison `LITERATURE_HIGH` query (see [`edison_run_followup_spatial.py`](edison_run_followup_spatial.py); raw artifacts in [`edison_artifacts/llm_cad_spatial_mitigation.*`](edison_artifacts/) — `status=success`, 33 reference contexts).

The goal is a concrete, citable answer to: *given that current frontier models (incl. Claude Opus 4.7) make characteristic spatial-reasoning and assembly mistakes when generating code-based CAD, what mitigations should we adopt for this repo, and where does manual intervention belong?*

## 1. Failure modes empirically observed in PR #35

The CadQuery model in `design/cad/single-channel-module/` exhibits the following defect classes, all of which are documented in the literature as characteristic LLM-CAD failure modes:

| # | Failure (PR #35 evidence) | Class | Literature parallel |
|---|---|---|---|
| F1 | Mounting bosses appear separate from the tap-collar ring; solenoid bracket "would probably break off easily" — not Boolean-unioned with the host body ([disc r3228441106, r3228681184](https://github.com/vertical-cloud-lab/powder-doser/pull/35)) | *Topological hallucination* — geometry exists but isn't attached to the assembly graph | Hallucinated/missing features and lack of constraint preservation are the headline limitations called out by `schupbach2025fromtextto pages 1-2`, `dong2025histcadgeometricallyconstrained pages 2-3`, and `xu2024brepgenabrep pages 9-11` (BrepGen reports missing faces, self-intersections, non-watertight outputs). |
| F2 | `make_corner_post()` drills the axial bore with `workplane(invert=True)`, likely cutting in the wrong direction (post may remain solid); `make_drivers()` uses `extrude(t, both=True)`, doubling thickness ([disc r3229031247, r3229031431](https://github.com/vertical-cloud-lab/powder-doser/pull/35)) | *Tool-API semantic errors* — operation runs without exception but produces wrong geometry | "Poor spatial reasoning [and] prompt sensitivity" identified by `schupbach2025fromtextto pages 1-2`. CAD-Coder explicitly motivates GRPO with **geometric and format rewards** specifically to catch this class (`guan2025cadcodertexttocadgeneration pages 1-3`). |
| F3 | `make_solenoid()` constructs a `plunger` solid but never unions it and returns only `body` ([disc r3229031292](https://github.com/vertical-cloud-lab/powder-doser/pull/35)) | *Dead code / forgotten union* — intent in the docstring not realised in geometry | Same family as F1; HistCAD's *editability benchmark* exists exactly because executable + visually-plausible geometry can still be structurally wrong (`dong2025histcadgeometricallyconstrained pages 12-12`). |
| F4 | "12 channels on a 150 mm pitch circle" gives ~39 mm spacing — 80×80 mm modules cannot fit ([disc r3229031398](https://github.com/vertical-cloud-lab/powder-doser/pull/35)) | *Numerical inconsistency* — claims that fail elementary arithmetic | LLMs over-confidently emit dimensions without solving the underlying constraint. Constraint-aware datasets (SketchGraphs, HistCAD) and solver-in-the-loop systems (`daareyni2025generativeaimeets pages 1-2`) target this directly. |
| F5 | No place for powder to enter the system; motor mounted on the auger entrance blocking loading ([disc r3228543667, r3228854193](https://github.com/vertical-cloud-lab/powder-doser/pull/35)) | *Functional-path omission* — geometry is plausible but the system can't perform its function | The literature on agentic CAD (`ocker2503fromideato pages 1-3`, `schupbach2025fromtextto pages 1-2`) explicitly recommends a **requirements / QA agent** that checks the design against the functional brief, not just geometry. |
| F6 | Vendor-part envelopes from PR #25 not reflected ([disc r3228476410](https://github.com/vertical-cloud-lab/powder-doser/pull/35)) | *Part-library hallucination* — model invents component dimensions | A core motivation for verified component libraries (the mechanical analogue of KiCad footprint libs); also the gap that retrieval-augmented CAD-Recode-style pipelines (`rukhovich2025cadrecodereverseengineering pages 4-6`) try to close. |
| F7 | "Made-up" 80 mm dispense-outlet-above-cup-rim clearance ([disc r3228811380](https://github.com/vertical-cloud-lab/powder-doser/pull/35)) | *Ungrounded numeric assumptions* — no derivation from operating envelope | `daareyni2025generativeaimeets pages 1-2` calls out prompt ambiguity and the need for solver/tool-supported derivation. |
| F8 | Unspecified fastener strategy (self-tap into FDM print vs. heat-set inserts) ([disc r3228875238](https://github.com/vertical-cloud-lab/powder-doser/pull/35)) | *DfAM omission* — manufacturable-by-default rules not encoded | DfAM agents and design-rule checks are highlighted as essential by `schupbach2025fromtextto pages 1-2`. |
| F9 | "Top-plate / bottom-plate / 4-posts" assembly chosen without considering 3D-print friendliness ([disc r3228660546](https://github.com/vertical-cloud-lab/powder-doser/pull/35)) | *Topology / archetype bias* — defaults to generic mechanical motifs rather than print-native ones | Lack of DfAM priors is a recurring complaint; `daareyni2025generativeaimeets pages 1-2` explicitly frames CAD generation as a *manufacturing workflow*, not a graphics task. |
| F10 | Broken doc cross-references to non-existent paths ([disc r3229031121, r3229031183, r3229031341](https://github.com/vertical-cloud-lab/powder-doser/pull/35)) | *Documentation hallucination* — mirrors the geometry hallucination but in prose | Text-side analogue of the same root cause; trivially caught by a link checker in CI. |

The pattern is consistent: **the model produces locally plausible code that compiles and renders, but the assembled artefact violates topological, numerical, functional, or manufacturability invariants that no test currently checks.**

## 2. Mitigation strategies, grouped

### A. Tool-side guardrails (deterministic verifiers run on the geometry the LLM produced)

Catch the failure modes that the model itself is structurally bad at noticing. Run on every CI commit and every `cad_model.py` regeneration.

1. **Watertightness / manifold check** — every printed part exported to STL must pass an OCCT/Open3D/`trimesh` manifold check. Catches F1, F3.
2. **Connected-components check on the union** — for each printed part, the final solid must be a single connected component. A floating boss or unattached bracket shows up as `n_components > 1`. Catches F1, F3 directly.
3. **Interference / clash detection on the assembly** — pairwise OCCT volume-of-intersection > ε on all moving parts (rotor vs collar) and > 0 on static parts that should mate. Catches F2-style wrong-direction cuts (the bore that didn't actually cut), F4 overlapping modules, and unintended interferences.
4. **Bounding-box / envelope assertions for vendor parts** — a `vendor_envelope.py` module per part (NEMA-17 stepper, GT2 pulley, 6805ZZ bearing, 5 mm solenoid) defining min/max bounding boxes and mating-feature locations. The LLM-emitted placeholder must pass `assert envelope_matches(part)` before the assembly is accepted. Catches F6.
5. **Test-driven CAD assertions** — a `tests/test_cad_model.py` per module with PyTest cases on:
   - total volume / mass per part within tolerance band,
   - hole count and diameter on each face,
   - fastener-engagement length ≥ insert depth,
   - dispense-outlet ↔ cup-rim clearance derived from `tilt_angle_deg` and `cup_rim_height_mm`, not hard-coded (catches F7),
   - `n_modules * module_chord_at_pitch_radius ≤ 2π·r_pitch` (catches F4).
6. **Slicer-side printability check** — overhang angle, minimum wall thickness, support-volume budget via PrusaSlicer / OrcaSlicer CLI. Catches F8/F9 latent costs.

### B. Prompting / agent strategies (make the model catch its own mistakes)

1. **Critic–revisor loop with multi-view rendering fed back as image input** — the lesson from `schupbach2025fromtextto pages 1-2` ("automated visual feedback improved performance, especially with multimodal models") and `ocker2503fromideato pages 1-3`. After generating `cad_model.py`, render iso/front/top/side and exploded views, then ask the same (or a separate "QA") model: *"For each fastener, name the part it screws into and the part it captures. Trace the powder path from inlet to outlet. List every Boolean operation that you intended but did not write."*
2. **Test-driven CAD prompting** — provide the assertions in §A.5 *as part of the prompt* and require the model to write code that passes them. This converts geometric correctness from a "judgement" task (which LLMs are bad at) into a "make this test green" task (which they are better at).
3. **ReAct-style tool use with a CAD kernel in the loop** — let the model evaluate `cq.Compound.fuse([...]).IsValid()`, count connected components, and inspect bounding boxes between turns rather than at the end. CAD-Coder's GRPO setup (`guan2025cadcodertexttocadgeneration pages 1-3`) is the strongest published evidence that *executable validation per step* materially raises pass rate.
4. **Self-consistency over independently-sampled designs** — sample N candidate `cad_model.py` files at temperature > 0; only keep ones that pass §A guardrails; have the model pick among the survivors with rationales. Cheap relative to the cost of a failed print.
5. **Explicit functional-path checklist in the system prompt** — e.g., "Powder must enter the system, be conveyed by the helix, exit, and fall into the cup. For each of these steps name the geometry feature responsible." Catches F5.
6. **Vendor-part retrieval before generation** — RAG over `hardware/parts/*.yaml` (datasheet excerpts) so dimensions are quoted, not invented. Catches F6.

### C. Representation choices (fix the data the model writes)

1. **Decompose into small parts with explicit interface contracts.** Per the convention already memorised for this repo (`design/cad/<module-name>/cad_model.py` + `sketch_2d.py` + `stl/` + `renders/`), keep each printed part in its own `make_<part>()` function with a docstring stating: (i) what it mates to, (ii) the datum it references, (iii) the fasteners that pass through it. Forces the model to be explicit about assembly intent and gives the verifier a hook.
2. **Parametric skeleton / datum part.** A first-class `make_skeleton()` function whose only job is to define the assembly's reference geometry (pitch circle, axis lines, mounting datums). Every other part is positioned *relative to the skeleton*, never with absolute literal coordinates. Catches F4 by construction (you can't get to "12 modules at 150 mm pitch" without writing the skeleton equation that proves spacing).
3. **Heat-set-insert-first fastener convention.** Standardise on **M3×5×4 brass heat-set inserts** in all FDM parts (per the PR #35 review and per the *kicad schematics* memory pattern of explicitly preferring the print-friendly option). Encode in a `fasteners.py` helper so every screw hole is generated by `heat_set_insert_pocket(diameter=..., depth=...)`, never by an ad-hoc `cylinder().cut()`. Catches F8.
4. **Vendor envelope library.** A `design/cad/_vendor_envelopes/` package containing CadQuery models of every COTS part listed in #25, each with verified `mating_face`, `bolt_pattern`, and `bounding_box` attributes. The LLM never invents these.
5. **Don't ask the model to reason in absolute coordinates.** Prefer `Workplane(face).workplane(offset=...)` over `Workplane((x, y, z))`. Most F2-class failures (wrong-direction cut/extrude) come from absolute coordinate reasoning where the model loses track of orientation.

### D. Where manual intervention fits

The literature is explicit that fully-autonomous CAD for fabricated hardware is **not yet** reliable (`schupbach2025fromtextto pages 1-2`, `daareyni2025generativeaimeets pages 1-2`, `ocker2503fromideato pages 1-3`); supervised generation with structured human checkpoints is the current state of the art. For this repo, the recommended division of labour is:

| Stage | Done by | What the human is checking | Artifact required |
|---|---|---|---|
| 1. Requirements / functional brief | **Human** (issue / brainstorming.md) | "What does this device need to do?" — full operating envelope, including tilt angles, powder properties, throughput, COTS parts list. | A short numbered requirements list in the module README, treated as the source of truth the agent is held to. |
| 2. Skeleton + datum + interface contracts | **Human-authored, model-extended** | That `make_skeleton()` correctly encodes the geometry the requirements imply; that interface contracts between parts are stated. | Skeleton CAD + dimensioned 2D sketch (`sketch_2d.py`). |
| 3. Per-part `make_<part>()` bodies | **Model**, with §A guardrails on every commit | Nothing manual in the inner loop; the verifier rejects bad outputs. | Passing `tests/test_cad_model.py`. |
| 4. Multi-view + exploded renders | **Model produces, human reviews** | Functional paths (powder inlet → outlet), fastener access, mating-face contact, vendor-part envelopes. | iso/front/top/side SVG + PNG + an *exploded* view + a *cross-section showing the powder column*. |
| 5. Print + dry-fit | **Human** | Anything the digital verifier can't see: real-world clearances, friction fits, snap features, FDM warpage, support scarring. | Photos / notes back into the README and into a `lessons.md` that future prompts can be conditioned on. |
| 6. Parameter changes (e.g., new motor, larger hopper) | **Human edits parameters; model regenerates** | That the parametric edit propagated correctly (HistCAD-style editability — `dong2025histcadgeometricallyconstrained pages 2-3`). | Re-run §A guardrails on the edited model. |

Manual intervention is **not** a fallback for when the model fails — it is **structurally required** at stages 1, 2, 5, and 6 because those stages encode information that is not in any training set: the device's actual operating envelope, the fabricator's process knowledge, and what broke last time.

## 3. Concrete recommendations for this repo

In priority order, addressing the highest-value mitigations first:

1. **Add `tests/test_cad_model.py` to every `design/cad/<module>/` package** with the assertions from §A.5 — connected-components, watertightness, vendor-envelope match, derived-clearance check, pitch-circle-fits check. Wire into the existing CI. *(Closes F1, F3, F4, F7 by construction.)*
2. **Introduce `design/cad/_vendor_envelopes/` and require every vendor placeholder to import from it.** *(Closes F6.)*
3. **Standardise on heat-set inserts in a `fasteners.py` helper.** *(Closes F8.)*
4. **Add an automated link-checker (`lychee` or `markdown-link-check`) to CI.** *(Closes F10, ~5-line CI change.)*
5. **Adopt a critic loop in the agent's CAD-generation workflow** — after `cad_model.py` is written, render iso/front/top/side + exploded + powder-flow cross-section and feed back into the model with the §B.1 / §B.5 prompts before opening the PR. *(Mitigates F2, F5, F9.)*
6. **Promote `make_skeleton()` and interface-contract docstrings to the `design/cad/<module-name>/` convention** documented in the *cad module convention* memory.

These together convert almost every defect catalogued in §1 from "caught by a human reviewer in PR" into "caught deterministically by CI before the PR is opened", which is the only sustainable path given the agent-driven workflow this repo is exploring.

## 5. Findings from the follow-up Edison query

The follow-up `LITERATURE_HIGH` query (raw outputs in [`edison_artifacts/llm_cad_spatial_mitigation.*`](edison_artifacts/)) returned 33 reference contexts spanning 2024–2026 work that directly target the failure modes catalogued in §1. The evidence sharpens the recommendations in §2 in five specific ways:

1. **Multi-agent + independent visual judge is the strongest published mitigation.** **CADSmith** (Barkley et al., 2026, `barkley2603cadsmithmultiagentcad pages 1-2`, `pages 6-7`) decomposes generation across Planner / Coder / Executor / Validator / Refiner agents with an independent VLM Judge that sees code, OCCT metrics, and three-view renders, achieving **100% execution success, median IoU 0.9629, mean Chamfer Distance reduced from 28.37 to 0.74 (~38×)**. Crucially, an ablation that removes the rendered three-view image *increases Tier-3 mean Chamfer Distance from 1.42 to 49.68* — empirically confirms §2.B.1 (visual feedback is necessary, not just helpful).
2. **Code-level checker feedback into the training loop works.** **CADCrafter** (Chen et al., 2025, `chen2025cadcraftergeneratingcomputeraided pages 6-7`) fine-tunes via DPO with code-checker feedback as the preference signal, reaching **Invalid Rate 3.6%** on DeepCAD multi-view. **GenCAD-Self-Repairing** (Tsuji et al., 2025, `tsuji2025gencadselfrepairingfeasibilityenhancement pages 5-7`) uses latent-space diffusion guidance to repair **65.84% of previously infeasible B-reps** (532/808), pushing feasibility from 93.1% → 97.0%. Both motivate adding §2.A's deterministic verifiers as a *training-time signal*, not just a CI gate.
3. **Connector-first design avoids late-assembly spatial reasoning.** **ArtiCAD** (Shui et al., 2026, `shui2026articadarticulatedcad pages 3-5`) explicitly argues that asking the model to infer assembly relationships *after* generating all part code is the highest-failure step; instead it predicts connectors (attachment points, local frames, semantic labels, joint limits) **at design time**, then assembly reduces to deterministic alignment with a constraint solver. This is exactly the *parametric skeleton + interface contracts* recommendation in §2.C.1–.2 — the literature gives it an evidence-backed name and validates the approach.
4. **Self-correction by the model alone is unreliable.** **CADReview** (Chen et al., 2025, `chen2025cadreviewautomaticallyreviewing pages 8-9`) finds **GPT-4o achieves only 41.5% accuracy** on detecting human-made errors in OpenSCAD programs, and closed-source MLLMs produce invalid OpenSCAD at **>25%** rate. The Känsälä thesis warns that multi-agent loops "converge prematurely, with most runs terminating after a single iteration or stopping at the iteration limit without resolving the identified issue" (`kansala2026generativeaidrivensystems pages 59-62`). Implication: §2.B's critic loops cannot replace §2.A's deterministic checks — they need to be paired.
5. **Code-level human edits are the highest-information correction signal.** **Sadik & Bujny** (2025, `sadik2509humanintheloopquantitativeevaluation pages 4-5`, `pages 1-2`) systematically compare correction modalities (orthographic views, sketches, image-only prompts, code-level edits) using volumetric similarity, surface similarity, Hausdorff distance, and ICP alignment, finding that **code-level correction achieves perfect or near-perfect metrics including zero Hausdorff distance**, dramatically outperforming image-only inputs. This validates the §2.D recommendation that *manual edits feed back into the parametric source*, not into rendered geometry — and tells us the automation ceiling for the §2.D table.

**Refined recommendations.** The §3 priority list is unchanged but reinforced. Two additions worth promoting:

- *Add a "Judge" agent step* (CADSmith-style) that consumes generated code + OCCT metrics + multi-view renders, separate from the model that wrote the code, before the PR is opened. This is the single highest-leverage intervention in the literature.
- *Predict the assembly graph (connectors + interface contracts) before writing per-part bodies* (ArtiCAD-style). Concretely: a `make_skeleton()` function whose output explicitly enumerates `(part_a, face_a, part_b, face_b, fastener)` tuples that the per-part `make_<part>()` functions are required to honor. This converts the assembly step from open-ended LLM reasoning to deterministic constraint solving.

The follow-up evidence shifts the manual-intervention table in §2.D very little: stages 1, 2, 5, and 6 remain human-required; the *cost* of stages 3–4 drops as multi-agent + judge architectures mature, but engineering-grade hardware still benefits from a code-level human review of the parametric source (Sadik 2025).
